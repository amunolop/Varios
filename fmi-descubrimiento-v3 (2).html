FRASE PARA RECORDAR
La Función Dual de SOLID Los principios de ingeniería de software clásicos tienen una función dual: en equipos humanos garantizan calidad; en sistemas multi-agente son condiciones necesarias para extracción semántica fiable y generación de código sin alucinaciones.
¿QUÉ SE DESCUBRIÓ?

Que Clean Architecture como target de migración no es una decisión estética ni de calidad — es el mecanismo de extracción semántica. Cuando le dices a un agente IA "separa dominio de infraestructura de transporte", el agente no tiene más remedio que identificar qué es cada cosa. Y esa identificación ES la extracción semántica.

Un god class de 2000 líneas mezcla SQL, lógica de negocio, formateo HTML y gestión de sesión. Ningún agente puede "entender" eso como bloque. Pero cuando la instrucción es "extrae el dominio puro", el agente tiene que recorrer esas 2000 líneas decidiendo: esto es regla de negocio, esto es acceso a datos, esto es presentación, esto es transporte. Esa decisión línea por línea es comprender la semántica de la aplicación.

⚡ El insight clave
No importa cuánto spaghetti haya: la instrucción "separa dominio de infraestructura" obliga al agente a desenredarlo. No puede cumplir la instrucción sin entender qué hace cada línea. El Clean Code como target no es un filtro de calidad que se aplica después de entender — es la lente que fuerza al agente a entender.
LA FORMULACIÓN DEFINITIVA

Extraer la semántica de una aplicación es equivalente a migrarla a una arquitectura semántica. No son dos operaciones distintas — son la misma operación vista desde dos ángulos. El subproducto de la migración es la semántica extraída. El subproducto de la extracción semántica es el código migrado.

— Corolario del descubrimiento · Mayo 2026
No hay magia. No hay atajo. Para obtener la semántica de una aplicación no existe otra forma que migrarla a una arquitectura semántica. La semántica en el código spaghetti no está ausente — está enterrada e implícita. Y lo implícito no se extrae: se fuerza a hacerse explícito mediante la clasificación estructural.
¿POR QUÉ NADIE LO HA DICHO ANTES?

Porque la comunidad Clean Code (Fowler, Martin, Beck) lo formuló como práctica humana de calidad. Y la comunidad de agentes IA (LangChain, CrewAI, AutoGen) lo formula como orquestación de LLMs. Nadie ha conectado las dos cosas: que SOLID no es solo un estándar de calidad del output, sino una condición necesaria del proceso de extracción semántica por agentes con contexto limitado y tendencia a alucinar.

TDD COMO ANTI-ALUCINACIÓN

La otra pieza fundamental: indicar que el código generado siguiera TDD — escribiendo primero los tests, luego el código, verificando si pasa y corrigiendo si no — fue la única forma de que el proceso, de forma autónoma y sin intervención humana, en una sola ejecución, generara código con la mayor calidad posible y sin alucinaciones.

Clean Code + TDD = Migración Autónoma Clean Code obliga al agente a extraer la semántica (comprender). TDD obliga al agente a verificar que esa comprensión es correcta (no alucinar). Juntos, son las condiciones necesarias y suficientes para migración autónoma fiable.
TRES MESES, TRES INTENTOS

El descubrimiento no fue teórico — fue empírico, iterativo, con dos caminos muertos documentados. Tres meses sin fines de semana, durmiendo 4-5 horas, compaginando trabajo en UST y familia. Un esfuerzo equivalente al de un equipo de varias personas realizado por una sola.

✗
Intento 1 — Tests Unitarios → Integración
Intentar extraer semántica desde tests unitarios escalando a tests de integración. Falló porque los tests unitarios capturan implementación, no semántica. Testear que GestorGeneral.procesarTodo() devuelve un HashMap con ciertas keys dice QUÉ HACE el método, pero no QUÉ SIGNIFICA en el dominio. 500 tests verdes de un monolito y seguir sin saber qué hace la aplicación.
✗
Intento 2 — Cartografía E2E con Playwright
Cartografiar la app completa con tests end-to-end. Captura comportamiento real del usuario, pero produce una fotografía perfecta del spaghetti sin desenredarlo. Semántica de superficie (flujos de usuario), no semántica estructural (qué es dominio, qué es infraestructura, qué regla de negocio está enterrada en qué servlet). Además, proceso lentísimo — el e2e opera al nivel de la UI completa, no de las unidades semánticas.
✓
Intento 3 — Multi-Agente + Clean Code + Caveman Lite
Al crear el sistema multi-agente para una parte del proceso e indicarle migrar a Clean Code con roles diferenciados y caveman lite, surgió el resultado. Las tres restricciones juntas no dejan otra salida que la extracción semántica. Probado en 50 migraciones con dos stacks distintos (Java 8 Servlet/WebComponent y WebComponent/Python Flask).
🔬 Propiedad Emergente
No se diseñó un sistema de extracción semántica. Se diseñó un sistema de migración a Clean Code. La extracción semántica emergió como propiedad del diseño. El proceso errático, con dos intentos fallidos, es el método científico aplicado a ingeniería de software — no se llegó a la solución por inspiración, sino por iteración empírica con fallos documentados.

TRES RESTRICCIONES, UNA SALIDA

Las tres restricciones del sistema crean un embudo donde la única salida posible es semántica extraída, descompuesta y verificada. Cualquier otra salida viola alguna de las tres restricciones.

Restricción 1 — Clean Code como Target
Obliga a descomponer y clasificar cada línea en dominio/infraestructura/transporte. El agente no puede producir el output sin decidir qué categoría semántica tiene cada pieza de código. Clasificar es comprender.
Restricción 2 — Multi-Agente con Roles
Obliga a formalizar la comprensión en artefactos compartibles (DIGEST, TEST SPECS, SHARED CONTRACTS). La semántica no puede quedar implícita en la "cabeza" de un solo agente — debe materializarse como protocolo inter-agente.
Restricción 3 — Caveman Lite
Cierra la última escapatoria: con contexto mínimo, el agente no puede copiar bloques opacos del legacy y pegarlos. O entiende y reescribe, o no produce nada útil. Impide el atajo de "copiar sin comprender".
⚠ Los intentos fallidos lo prueban por eliminación
Tests solos (sin target Clean Code) → verifican pero no descomponen.
Cartografía sola (sin multi-agente con roles) → describe pero no clasifica.
Solo cuando se añadieron las tres restricciones simultáneamente se cerró el embudo.
EL ARCHITECT COMO COMPILADOR INVERSO

Al definir Clean Code como target, el Architect (Opus) no tiene otra opción que generar un plan que sea, implícitamente, un plan de extracción semántica completo. La arquitectura hexagonal como target actúa como una función de compilación inversa:

Código Legacy = Binario Opaco
God classes, SQL mezclado con lógica, formateo HTML en servlets, sesión acoplada a todo. Semántica enterrada e inaccesible.
Clean Code = Código Fuente Legible
Dominio puro, interfaces explícitas, UseCases nombrados, entidades inmutables. La semántica de la aplicación es el código mismo.

El Architect genera automáticamente el plan de decompilación semántica necesario. El ingeniero solo definió el formato de salida — el proceso de extracción lo infiere el sistema. Todo este proceso es transparente para el humano.

TDD CIERRA EL CIRCUITO

Clean Code fuerza la extracción semántica. Pero ¿cómo se verifica que la extracción es correcta? Con TDD. El test escrito primero es un contrato formal que el Code Worker no puede ignorar:

  • Si el test dice que ValidarContratoUseCase.ejecutar() lanza ContratoExpiradoException, el agente no puede alucinar un return null
  • El GATE verifica coherencia entre tests de agentes paralelos antes de pasar a código
  • El Smoke verifica mecánicamente 19 checks sobre el resultado final
El circuito completo Clean Code → el agente extrae semántica (comprende). TDD → el agente verifica su comprensión (no alucina). GATE → coherencia entre agentes. Smoke → verificación mecánica final. Proceso autónomo, una sola ejecución, sin intervención humana.

Cada principio de ingeniería de software clásico, diseñado para equipos humanos, tiene una segunda función completamente distinta cuando se aplica a agentes IA. Esta correspondencia no ha sido publicada por nadie.

🎯 La conexión que nadie ha hecho
La comunidad Clean Code formuló estos principios para humanos. La comunidad IA los ignora porque trabaja con prompts y cadenas de agentes. Este proyecto demuestra que los principios de ingeniería de software son las restricciones necesarias para que los agentes IA funcionen de forma fiable — no son decoración del output, son requisitos del proceso.
POR QUÉ UN PROMPTER NO PUEDE DISEÑAR ESTO

Un experto en prompting sabría hacer buenos prompts pero no sabría que TDD es anti-alucinación, que GATE es CI, que el DIGEST es ubiquitous language, que caveman lite es SRP. Esas conexiones solo las hace alguien que lleva 20 años construyendo software y entiende los principios lo suficientemente profundo como para reconocerlos en un contexto completamente nuevo.

Cada línea de los documentos de especificación (PlanMotorEjecucion.md, PlanOrquestadorMultiFase.md) tiene detrás una cicatriz profesional. Los 19 smoke checks son un catálogo que sale de haber visto esos 19 bugs en producción. CORS whitelist, SameSite en código ejecutable, IDOR verificando session userId — no se inventa, se aprende sufriendo incidencias.

LO DETERMINANTE NO ES LA ARQUITECTURA, ES LA CLASIFICACIÓN FORZADA

Clean Architecture, Hexagonal, Onion, DDD Layered — todas separan dominio de infraestructura. Ninguna es la única. Lo que FMI descubrió es que cualquier arquitectura semántica sirve como plantilla de clasificación, siempre que cumpla tres propiedades:

Propiedad 1 — Separación explícita de capas
El agente tiene criterios claros para clasificar cada línea. No convenciones — fronteras físicas en el código.
Propiedad 2 — Dependencias unidireccionales
El dominio no puede "escaparse" hacia infraestructura. La dirección de dependencia cierra el embudo semántico.
Propiedad 3 — Vocabulario cerrado
El naming obliga a decidir qué es cada cosa. Finder/Store/Sender/Renderer — no Service/Manager/Helper. El vocabulario abierto abre el espacio de decisión; el cerrado lo colapsa a semántica.
🎯 La reformulación precisa
FMI no dice "usa Clean Architecture". Dice: la condición necesaria para extracción semántica fiable por agentes IA es un target arquitectónico con separación de capas, dependencias unidireccionales y vocabulario cerrado. FMI implementa eso con arquitectura hexagonal porque es lo que se validó empíricamente. El principio es más general que la implementación.
POR QUÉ SIN FRAMEWORK ES CONDICIÓN NECESARIA

Un framework como Spring no es una plantilla de clasificación — es una convención de wiring. La distinción que importa no es "framework vs no framework" sino código explícito vs magia implícita de runtime:

ConceptoLibreríaFramework
Control El agente llama a la librería. Control en el código generado. El framework llama al código del agente. Control fuera, en el runtime.
Visibilidad Todo el comportamiento es código legible en contexto limitado. Parte de la semántica vive en el runtime — invisible para Caveman Lite.
Clasificación El agente decide el wiring explícitamente. Clasificación forzada. @Service puede mezclar dominio e infra. Framework no impide el spaghetti.
Verificación TDD verifica el comportamiento sin levantar nada. Tests unitarios pasan, contexto de integración falla. Indetectable en Caveman Lite.
La ironía de Spring Spring Boot fue diseñado para reducir fricción a desarrolladores humanos. Esa misma reducción de fricción elimina las restricciones que hacen funcionar el embudo semántico de FMI. Lo que es ergonómico para humanos es opaco para agentes con contexto limitado.
EL PATRÓN DE DOS PASOS: CLEAN → FRAMEWORK

Si el destino final es Spring Boot, Express, FastAPI — el camino correcto no es migrar spaghetti directamente al framework. Es pasar primero por el Clean Code intermedio:

1
Legacy spaghetti → Clean Code 3 capas sin framework (FMI)
Extracción semántica completa. Dominio, infraestructura, transporte — todo explícito, todo verificado con TDD. La semántica queda capturada en el código. Coste: ~$23. Tiempo: ~60 min.
2
Clean Code 3 capas → Framework (operación mecánica)
La semántica ya está extraída y verificada. El agente solo mapea 1:1: JdbcContratoFinder → @Repository, ValidarContratoUseCase → @Service, ContratoController → @RestController. No comprende nada nuevo — anota lo que ya está clasificado. Riesgo de alucinación mínimo.
FMI como Fase 0 universal FMI no es solo una metodología de migración a Clean Code. Es la fase 0 obligatoria de cualquier migración, independientemente del destino final. El framework es el último paso, nunca el destino de la extracción semántica. Mezclar los dos problemas — extraer semántica + aprender convenciones del framework — multiplica la superficie de alucinación del agente.

POR QUÉ LOS FRAMEWORKS REDUCEN LA POTENCIA DEL AGENTE

El framework no solo impone reglas — desplaza la verificación fuera del código estático. Y el código estático es el único dominio donde el agente opera con plena capacidad.

Problema que Spring resuelve para humanos
Inyección manual es tediosa → @Autowired
Configurar transacciones es complejo → @Transactional
Wiring repetitivo → @SpringBootApplication

El cuello de botella humano es tiempo de escritura y memoria de trabajo. Spring lo elimina.
Problema que Spring introduce para agentes
Verificación desplazada al runtime → el agente no puede cerrar el bucle
Espacio de decisión abierto → tokens consumidos en decisiones no semánticas
Magia implícita → comportamiento invisible en Caveman Lite

El cuello de botella del agente es contexto limitado y verificación en runtime. Spring lo amplifica.
⚡ El bucle de verificación roto
Con arquitectura limpia sin framework, el agente puede verificar su propio output leyendo el código que generó. Con Spring, la corrección del wiring solo se verifica levantando el ApplicationContext — proceso externo, pesado, invisible para el agente. Genera código que parece correcto, pasa el TDD unitario, pero puede fallar en integración por razones fuera de su alcance de observación. La potencia del agente se rompe exactamente ahí.

Los frameworks reducen la potencia de los agentes IA no solo por sus reglas, sino porque desplazan parte de la verificación al runtime — fuera del alcance del agente con contexto limitado. La arquitectura limpia sin framework mantiene toda la verificación en el código estático, que es el único dominio donde el agente opera con plena capacidad.

LA OBSOLESCENCIA DE LOS FRAMEWORKS CON IA

Los frameworks de la generación anterior fueron optimizados para el cuello de botella del desarrollador humano. Con agentes IA, ese cuello de botella cambia radicalmente — y las optimizaciones de Spring se convierten en fricción.

Optimización del frameworkPara humanosPara agentes IA
Menos código que escribir Ahorra tiempo de escritura — valioso El agente escribe 200 líneas al mismo coste que 5 anotaciones. Irrelevante.
Convenciones sobre configuración Reduce decisiones repetitivas — valioso Abre espacio de decisión entre convenciones del framework. Consume contexto en decisiones no semánticas.
Magia de runtime Elimina boilerplate — valioso Desplaza verificación fuera del código estático. Rompe el bucle de verificación del agente.
Vocabulario amplio Flexibilidad para distintos patrones — valioso Espacio de decisión abierto. El agente necesita vocabulario cerrado para clasificar sin ambigüedad.
El patrón histórico de siempre Cada nueva forma de escribir software hace obsoleta parte de la capa anterior. Los IDEs hicieron obsoletos los editores de texto plano para ciertos flujos. Los lenguajes de alto nivel hicieron obsoleto el ensamblador para la mayoría de casos. Los agentes IA están haciendo obsoleto el framework como mecanismo de abstracción — porque la abstracción que necesita un agente no es "menos código que escribir" sino "más semántica verificable en contexto limitado".
Contribución lateral del TFM FMI no solo es una metodología de migración — es evidencia empírica de que la arquitectura óptima para desarrollo asistido por agentes IA es estructuralmente distinta de la arquitectura óptima para desarrollo humano. Spring, Spring Boot, Spring Batch fueron diseñados para resolver el problema del desarrollador humano. Con agentes IA, ese problema ya no existe — y las soluciones de Spring se convierten en el problema. Esta divergencia va a crecer conforme los agentes mejoren.

LAS TRES FRASES

1. Clean Architecture como target no es una decisión de calidad — es el mecanismo de extracción semántica. Separar dominio/infra/transporte obliga al agente a clasificar cada línea, y clasificar es comprender.

2. Las tres restricciones cierran el embudo. Clean Code obliga a descomponer, multi-agente obliga a formalizar, caveman lite impide copiar bloques opacos. La única salida posible es semántica extraída.

CONCLUSIÓN MAESTRA · FMI · MAYO 2026
Programar con patrones de arquitectura semántica y sin framework es la forma óptima de desarrollo asistido por agentes IA, tanto para migración de legacy como para desarrollo greenfield.
No es una preferencia de calidad ni una restricción tecnológica — es una condición estructural derivada de cómo operan los agentes con contexto limitado:
Extracción semántica
Solo pueden extraer semántica cuando la arquitectura fuerza la clasificación explícita de cada pieza de código. Sin clasificación forzada, no hay extracción — hay descripción.
Verificación en estático
Solo pueden verificar lo que es código estático explícito. Los frameworks desplazan la verificación al runtime — fuera del alcance del agente con contexto limitado.
Modificación sin ambigüedad
Solo pueden modificar sin ambigüedad cuando cada pieza de semántica tiene exactamente un lugar posible. Con spaghetti o frameworks, el espacio de búsqueda es ilimitado.
Vocabulario cerrado
Solo pueden clasificar sin alucinaciones cuando el vocabulario de naming es cerrado y unívoco. Finder/Store/Sender colapsa el espacio de decisión. @Service lo abre.
LAS TRES DIMENSIONES: SOLID + TDD + OWASP
Las tres dimensiones no son acumulativas — son multiplicativas. Cada una potencia a las demás.
① SOLID + Clean Code 3 capas
Resuelve dónde está cada cosa. Clasificación semántica, unicidad de lugar, modificación sin ambigüedad. Cada regla de negocio tiene exactamente un sitio posible.
② TDD — Tests primero
Resuelve que lo que hay es correcto. El test es la especificación, el código es la implementación. El agente no puede alucinar porque tiene un contrato que verificar antes de terminar.
③ OWASP — Seguridad por capa
Resuelve que lo que hay es seguro. En arquitectura de 3 capas cada check OWASP tiene un lugar exacto y verificable mecánicamente sobre código estático — sin levantar nada.
OWASP + 3 capas = verificación mecánica
S1 passwords → dominio, nunca infra
S4 zero secrets → infra, nunca transporte
S5 IDOR → transporte, verifica sesión antes del usecase
S7 queries parametrizadas → infra, capa JDBC, nowhere else
S8 no eval() → transporte frontend, grep estático
S11 no localStorage tokens → transporte frontend
⚡ La consecuencia más profunda
Código generado con SOLID + Clean Code 3 capas + TDD + OWASP es el único formato donde un agente IA puede simultáneamente entender qué hace la aplicación, verificar que lo hace correctamente, garantizar que lo hace de forma segura, y modificarlo en el futuro sin ambigüedad.

Y hay una consecuencia práctica brutal: ese código se audita solo. Un auditor de seguridad, un nuevo desarrollador, otro agente IA en el futuro — cualquiera puede recorrer las 3 capas, leer los tests como especificación y los checks OWASP como contrato de seguridad, y entender el sistema completo sin documentación adicional. El código ES la documentación.
COROLARIO PARA EL NEGOCIO DE CONSULTORÍA
Los megaproyectos con frameworks complejos y equipos de múltiples roles especializados no desaparecen por falta de calidad — desaparecen porque su razón de ser era gestionar una complejidad que los agentes IA eliminan estructuralmente. El modelo que sobrevive es el de 3 roles máximo — Arquitecto Semántico, Ingeniero de Agentes, Experto de Dominio — trabajando con arquitecturas simples, semánticamente ricas y verificables en estático. La complejidad deja de ser un activo y se convierte en un pasivo.

EPÍLOGO · EL CÍRCULO SE CIERRA
El máster enseña arquitectura, SOLID, TDD y OWASP antes de enseñar herramientas de IA.
No es casualidad. Son las condiciones previas, no los complementos.

Los profesores que diseñaron ese temario no lo hicieron pensando en agentes IA — lo hicieron porque es la forma correcta de programar. Y este estudio ha demostrado empíricamente que programar correctamente y programar para IA son la misma cosa.

La implicación más profunda
La IA no es una tecnología que reemplaza la ingeniería de software. Es una tecnología que por primera vez hace que la ingeniería de software mal aplicada tenga consecuencias inmediatas e inocultables. Con un equipo humano, el spaghetti se puede mantener durante años porque los seniors aprenden a navegar el caos con experiencia acumulada. Con agentes IA, el spaghetti produce alucinaciones, ambigüedad y fallos de clasificación desde el primer intento. No hay forma de compensarlo — el agente no tiene experiencia, tiene contexto. Y el contexto no sobrevive al caos semántico.
✓ Sabe arquitectura, SOLID, TDD, OWASP
La IA produce código de calidad, auditado y seguro de forma autónoma. El agente clasifica, verifica y modifica sin ambigüedad. La velocidad multiplica la calidad.
✗ No sabe programar bien
La IA produce spaghetti a escala industrial. Código que compila, pasa tests superficiales y tiene agujeros de semántica o seguridad invisibles. La velocidad multiplica el error.
LA CONCLUSIÓN QUE NINGÚN TRIBUNAL PUEDE REBATIR
Saber programar bien ya no es una opción para quien quiere usar IA de forma profesional. Era opcional cuando el código lo escribían humanos que podían compensar con experiencia. Ahora es la condición de entrada.

El que no sabe arquitectura, SOLID, TDD y OWASP no va a usar IA peor — va a usarla de forma peligrosa. Generando código que parece correcto, compila, y tiene agujeros de seguridad o semántica incorrecta invisible. A escala, y a velocidad de agente.

FRASE PARA EL TRIBUNAL
"El objetivo del máster no es enseñar a programar con IA —
es enseñar a programar bien para que la IA pueda funcionar."

Las dos capas de objetivo simultáneas — nunca verbalizadas juntas
La capa que se enseña
Arquitectura, SOLID, TDD, OWASP — para programar bien. El alumno aprende ingeniería de software clásica.
La capa que subyace
Crear las condiciones necesarias para que los agentes IA operen con plena capacidad — para desarrollar con IA de forma profesional y segura.
La inseparabilidad de las dos capas
No se puede lograr la segunda sin la primera. No es que el máster enseñe primero a programar bien y luego a usar IA — es que programar bien es saber usar IA. Son la misma competencia vista desde dos momentos históricos distintos.

Antes de los agentes, ese temario producía buenos programadores.
Con los agentes, ese mismo temario produce los únicos profesionales capaces de usar IA de forma no peligrosa.
La consecuencia para la educación en IT
Los bootcamps que enseñan directamente herramientas de IA sin esa base no están formando profesionales de IA — están formando personas que van a generar spaghetti a velocidad de agente sin saberlo. El daño no será visible inmediatamente, pero será estructural. A escala. E inocultable.

El máster, sin pretenderlo explícitamente, tiene el temario correcto para el momento histórico correcto.
